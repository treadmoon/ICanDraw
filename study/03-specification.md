# 第 3 章：编写功能规格

> 核心问题：怎么把需求写到"没有歧义"的程度？

---

## 3.1 从产品定义到功能规格

上一章定了"底线"（宪法），再上一章定了"方向"（产品定义）。但这两样东西都不能直接让 AI 写代码——它们太抽象了。

功能规格（Specification）就是把抽象的产品定义翻译成具体的、可验收的需求文档。它回答的核心问题是：**"做完了长什么样？怎么判断做没做完？"**

Spec-Kit 用 `/speckit.specify` 命令生成功能规格。输入是一句话描述，输出是结构化的 `spec.md`。

ICanDraw 的输入：

> "ICanDraw Phase 1 — AI 图表生成器 MVP，对话式生成 ECharts 图表 + Excalidraw 手绘批注到画布上"

## 3.2 User Story：用优先级讲故事

功能规格的核心是 User Story（用户故事）。每个故事描述一个完整的用户场景，并标注优先级。

ICanDraw 的 4 个用户故事：

| 优先级 | 故事 | 核心价值 | 为什么是这个优先级 |
|--------|------|---------|------------------|
| P1 | 对话生成图表 | 产品核心链路 | 没这个，产品不存在 |
| P2 | AI 手绘批注 | 与普通 BI 的核心差异 | 这是"情绪化批注"的卖点 |
| P3 | 对话迭代修改 | AI 原生产品关键体验 | 协作打磨，而非一次性生成 |
| P4 | CSV 数据上传 | 从玩具到工具的跨越 | 真实数据输入 |

**优先级排序的逻辑**：P1 是地基，没有它其他都不存在。P2 是差异化，没有它产品没有竞争力。P3 是体验升级。P4 是实用性补充。

这个排序直接决定了后面的开发顺序——先做 P1，P1 做完才能做 P2/P3/P4。

## 3.3 Given/When/Then：消灭歧义的武器

每个 User Story 都附带验收场景，用 Given/When/Then 格式：

**User Story 1 的验收场景**：

> **Given** 用户在对话框输入"画一个柱状图，展示周一到周五的销售额"
> **When** 按下回车
> **Then** 画布上出现一个柱状图，X 轴为周一到周五，Y 轴为 AI 生成的示例数据，图表支持 hover tooltip

为什么要这么写？因为**自然语言天然有歧义**。

比如"生成图表"——生成到哪里？对话面板里？新窗口？画布上？图表能不能交互？能不能拖动？

Given/When/Then 把每一步都钉死了：
- Given（前提）：用户做了什么操作
- When（触发）：什么事件发生
- Then（结果）：系统应该呈现什么状态

**测试的时候照着验就行，不用猜"这个功能到底算不算做完了"。**

## 3.4 边界情况：AI 最容易忽略的地方

spec.md 还定义了 Edge Cases（边界情况）。这些是 AI 写代码时最容易忽略的：

| 边界情况 | 期望行为 |
|---------|---------|
| 用户输入与数据可视化无关的内容 | AI 礼貌引导回数据叙事场景 |
| CSV 格式异常（编码错误、缺失值） | 明确的错误提示和修复建议 |
| 画布上有多个图表时，修改指令指向不明 | 通过上下文或点击选中确定目标 |
| AI 生成的 ECharts option 格式错误 | 前端优雅降级，显示错误信息而非白屏 |

如果你不在 spec 里写这些，AI 生成的代码大概率不会处理这些情况。它只会处理"正常路径"。

## 3.5 功能需求：用 MUST 关键字

spec.md 用 MUST 关键字定义了 8 条功能需求（FR-001 ~ FR-008）：

```
FR-001: 系统 MUST 提供左右分栏布局——左侧画布，右侧对话面板（可收起）
FR-002: 系统 MUST 将用户自然语言输入发送给 LLM，获取结构化 JSON 响应
FR-003: 系统 MUST 在画布上渲染 ECharts 图表实例，支持基本交互
FR-004: 系统 MUST 在画布上渲染 Excalidraw 手绘元素
FR-005: 系统 MUST 支持对话历史上下文
FR-006: [未实现] 系统 MUST 支持 AI 流式响应
FR-007: 系统 MUST 支持 CSV 文件拖拽上传和解析
FR-008: 系统 MUST 维护画布状态为 JSON 文档，支持 AI 增量修改
```

注意 FR-006 标注了"未实现"——这是后来代码实现阶段发现技术方案变更后回来标注的。spec 是活文档，不是写完就不动了。

## 3.6 成功标准：可量化的验收指标

| 标准 | 指标 |
|------|------|
| SC-001 | 用户输入后 3 秒内画布开始渲染 |
| SC-002 | 支持至少 5 种图表类型（折线、柱状、饼图、散点、面积图） |
| SC-003 | AI 批注至少包含 1 个数据洞察 |
| SC-004 | 对话修改成功率 > 80% |
| SC-005 | CSV（< 10MB）上传后 2 秒内完成解析 |

这些数字不是拍脑袋定的——它们直接影响了后面 plan.md 里的技术约束（"AI response < 3s"、"CSV parse < 2s for < 10MB"）。

## 3.7 假设：明确说出"我们假定什么"

spec.md 最后列出了 Assumptions（假设）：

- MVP 使用 GPT-4o 兼容 API 作为 LLM 后端
- 用户有现代浏览器
- 不需要登录
- 不需要数据持久化
- ECharts 和 Excalidraw 集成采用 embeddable 方案

假设的价值在于：**当假设不成立时，你知道哪些设计需要重新审视**。比如后来发现豆包模型和 Vercel AI SDK 不兼容，"使用 GPT-4o 兼容 API"这条假设就不成立了，于是 API 调用方案整个重写。

## 3.8 教学要点

1. **User Story 的优先级决定开发顺序**。不是所有功能同等重要。P1 做不完，P2~P4 都没意义。
2. **Given/When/Then 是消灭歧义的最佳格式**。如果你不能用这个格式描述一个功能，说明你还没想清楚。
3. **边界情况要在 spec 阶段就想到**，不要等代码写完了才发现"用户输入乱七八糟的东西怎么办"。
4. **成功标准要可量化**。"体验好"不是标准，"3 秒内开始渲染"才是。
5. **假设要显式写出来**。隐含的假设是最危险的——它们会在你最意想不到的时候崩塌。

## 3.9 动手练习

为你的项目写一个 User Story（P1 优先级），包含：

1. 一段自然语言描述
2. 至少 2 个 Given/When/Then 验收场景
3. 至少 1 个边界情况
4. 1 条可量化的成功标准
