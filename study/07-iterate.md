# 第 7 章：迭代与文档维护

> 核心问题：代码改了，文档怎么跟上？团队怎么协作？

---

## 7.1 MVP 之后才是真正的开始

前六章走完，ICanDraw 的 MVP 跑起来了。但产品不可能一次做完——MVP 之后经历了大量迭代：

| 迭代 | 改了什么 | 触发原因 |
|------|---------|---------|
| 布局翻转 | 画布从右移到左，对话面板从左移到右 | 用户反馈 |
| 去掉 AI SDK | 直接 fetch 替代 Vercel AI SDK | 豆包模型不兼容 |
| Overlay → Embeddable | ECharts 渲染方案重写 | 图表不能拖拽 |
| 数据规范化 | 新增 normalizeChart/normalizeAnnotation | AI 返回格式不一致 |
| 流程图支持 | 新增 Drawing 类型和 toExcalidrawElements | 用户需要画流程图 |
| 箭头修复 | 从 points 自动计算 width/height | 箭头错位 |
| 国际化 | 新增 i18n-store.ts | 用户要求 |

每一次迭代都改了代码，但**文档没有同步更新**。这导致了一个严重问题。

## 7.2 文档脱节：一个真实的教训

ICanDraw 在迭代过程中，plan.md、tasks.md、spec.md 都停留在最初的设计方案。具体有多少处不一致？

**plan.md 的 12 处过时描述**：
- 框架版本写的 Next.js 15，实际是 16
- 依赖列表里有 Vercel AI SDK，实际没用
- 文件结构里有 ChartOverlay.tsx，实际已删除
- 架构决策写的 overlay 方案，实际已改为 embeddable
- ……

**tasks.md 的 10 处不一致**：
- T009 描述 "Vercel AI SDK"，实际直接 fetch
- T013 描述 "ChartOverlay"，实际是 EChartEmbeddable
- T028/T029/T031 标记未完成，实际已完成
- 缺少 8 个实际完成的任务（布局变更、i18n、drawings 等）

**spec.md 的 3 处错误**：
- FR-001 布局方向写反了
- FR-006 流式响应标记为必须，实际未实现
- Assumptions 里的 overlay 方案已过时

**后果**：如果一个新人加入项目，读了这些文档，他会以为项目用的是 overlay + AI SDK 方案，然后写出和现有代码完全不兼容的东西。

**过时的文档比没有文档更危险。**

## 7.3 解决方案：/sync-docs 命令

ICanDraw 项目为此创建了一个 `/sync-docs` prompt（`.kiro/prompts/sync-docs.md`），手动触发时 AI 会：

1. 读取所有 spec 文档 + 所有源码
2. 逐维度对比：技术栈、文件结构、架构决策、任务状态、需求实现
3. 输出差异表，等待确认
4. 确认后逐文件更新

**设计原则**：

| 原则 | 理由 |
|------|------|
| 只追加不删除 | 原始任务编号不变，保持可追溯性 |
| 保留原文做对照 | 变更的决策加 `[Updated]` 前缀，保留原始描述 |
| 不确定的标记"需确认" | 不自作主张改东西 |
| 不改用户故事和验收标准 | 那些是原始需求，即使没实现也不该改 |

**为什么不用 hook 自动更新？**

考虑过三个方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| Hook 自动更新 | 全自动 | hook 只看到单条回复，缺乏全局视角，容易改错 |
| 手动触发 AI 同步 | AI 能看到完整代码和文档，更新质量高 | 需要人记得触发 |
| Git pre-commit 检测 | 提交时提醒 | 只能提醒不能修复，实现复杂 |

选了方案 B。plan.md/tasks.md/spec.md 是结构化规划文档，不适合被 hook 零散修改。

## 7.4 决策日志自动记录

虽然 spec 文档不适合自动更新，但**决策日志**适合。ICanDraw 配置了 Kiro CLI 的 stop hook：

**工作原理**：
1. 每次 AI 回复后触发 `.kiro/hooks/log-decisions.sh`
2. 脚本从回复中匹配决策关键词（改了/换成/删除/新增/因为...所以）
3. 匹配到就提取摘要，带时间戳追加到 `docs/decision-log.md`

**配置**（`.kiro/agents/default.json`）：
```json
{
  "hooks": {
    "stop": [{
      "command": "bash .kiro/hooks/log-decisions.sh",
      "timeout_ms": 5000
    }]
  }
}
```

这样每个技术决策都会被自动捕获，不需要手动记录。

## 7.5 文档体系就是 AI 的"外置记忆"

每次开新的 AI 对话，AI 对项目一无所知。你得重新告诉它技术栈、目录结构、之前的决策。

ICanDraw 的文档体系解决了这个问题——它是 AI 的"外置大脑"：

```
.specify/memory/constitution.md  ← AI 的"价值观"：什么能做什么不能做
specs/001-canvas-mvp/spec.md     ← AI 的"需求记忆"：用户要什么
specs/001-canvas-mvp/plan.md     ← AI 的"架构记忆"：技术方案是什么
specs/001-canvas-mvp/tasks.md    ← AI 的"进度记忆"：做到哪了
docs/decision-log.md             ← AI 的"经验记忆"：踩过什么坑
AGENTS.md                        ← AI 的"项目记忆"：技术栈、编码规范
```

AI 工具启动时自动读取这些文件。不管是谁、不管什么时候开对话，AI 都能"记住"项目全部背景。

**关键洞察：文档不是写给人看的附属品，而是 AI 的外置大脑。维护文档就是维护 AI 的记忆。**

## 7.6 迭代的三种规模

不是每次改动都需要走完整流程：

| 规模 | 典型场景 | 怎么做 |
|------|---------|--------|
| 小改动 | Bug 修复、样式微调 | 直接改代码，不更新文档 |
| 中等功能 | 新增独立能力（如 i18n） | 更新现有 spec 文档 + 写代码 |
| 大功能 | 独立模块（如多人协作） | 创建新 feature，走完整 Spec-Kit 流程 |

中等功能的更新顺序：
```
1. spec.md   → 追加 User Story
2. plan.md   → 更新目录结构、补充技术决策
3. tasks.md  → 追加新 Phase 和 Task
4. 写代码
5. 跑 /sync-docs 确认文档一致
```

## 7.7 团队协作：打破 AI 孤岛

一个人用 AI 编程很快，但如果一直是一个人干，会出现"AI 孤岛"——知识锁在一个人脑子里，代码没人能接手。

Spec-Kit 的文档体系天然是团队协作的接口：

```
宪法 (constitution.md)  ← 全员对齐的底线共识
  ↓
需求 (spec.md)          ← 产品和开发的沟通契约
  ↓
设计 (plan.md)          ← 技术方案的评审材料
  ↓
任务 (tasks.md)         ← 可分配、可并行的工作单元
  ↓
代码 (src/)             ← 可追溯、可接手的实现
```

**新人上手路径**：
```
1. constitution.md  → 5 分钟了解底线和技术栈
2. spec.md          → 15 分钟了解产品功能
3. plan.md          → 15 分钟了解技术架构
4. tasks.md         → 10 分钟了解开发进度
5. 挑一个未完成的任务 → 开始干活
```

不需要有人花半天时间"讲项目"。文档就是最好的 onboarding 材料。

**避免 AI 孤岛的 5 条实践**：

1. **文档先行，代码后行**——先更新 spec/plan/tasks，再让 AI 写代码
2. **关键决策写进文档**——AI 聊天记录会丢，文档不会
3. **每个提交关联任务编号**——`T012: 创建 ChatPanel 组件`，不写 `fix stuff`
4. **定期同步**——每天 10 分钟过一遍 tasks.md
5. **AI 生成的代码必须有人 Review**——两个人的眼睛比一个人加一个 AI 更可靠

## 7.8 全流程回顾

回顾 ICanDraw 从零到一的完整流程：

```
一句话想法："用户说话，AI 画图"
  │
  ▼ 第 1 章
产品定义迭代（spec.md → spec2.md → spec3.md）
  │
  ▼ 第 2 章
项目宪法（6 条不可违背的原则）
  │
  ▼ 第 3 章
功能规格（4 个 User Story + Given/When/Then 验收标准）
  │
  ▼ 第 4 章
实施计划（技术选型 + 目录结构 + 关键决策）
  │
  ▼ 第 5 章
任务拆解（7 个阶段、31 个任务、依赖关系、并行标注）
  │
  ▼ 第 6 章
AI 代码实现（18 个文件、~1000 行代码、4 个踩坑修复）
  │
  ▼ 第 7 章
迭代维护（8 个迭代任务、文档同步、决策日志自动化）
```

**核心结论**：AI 编程不是"不用想了"，恰恰相反——它要求你想得更清楚。因为 AI 不会帮你做判断，它只会执行你的指令。你的指令越精确，它的输出越靠谱。

花 70% 的时间在"想"上（第 1~5 章），30% 在"做"上（第 6~7 章）。这和传统开发正好反过来。

## 7.9 教学要点

1. **过时的文档比没有文档更危险**。代码改了就要同步文档，否则会误导后来的人。
2. **规划文档适合手动触发同步，决策日志适合自动记录**。两种文档的更新频率和粒度不同，用不同的策略。
3. **文档是 AI 的外置记忆**。维护文档就是维护 AI 的记忆，让每次对话都能站在同一个起点上。
4. **团队协作的核心是共享文档，不是共享聊天记录**。Git push/pull 就是记忆的同步机制。

## 7.10 最终练习

回顾你在前六章练习中产出的所有文档：

1. 产品定义（第 1 章）
2. 项目宪法（第 2 章）
3. User Story + 验收场景（第 3 章）
4. 技术决策记录（第 4 章）
5. 任务清单（第 5 章）
6. AI 生成的代码 + Review 记录（第 6 章）

现在假设你做了一次架构变更（比如换了一个依赖库），尝试：

1. 列出哪些文档需要更新
2. 每份文档具体改哪些内容
3. 写一条决策日志：做了什么变更、为什么、影响了哪些文件
